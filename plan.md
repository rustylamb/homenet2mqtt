1.  **Understand**
    *   The `api/config/raw/:type/:entityId` endpoint fetches configurations via `ctx.getCurrentRawConfigs()`.
    *   For `type === 'entity'`, it iterates over all configs and all entity types (`ENTITY_TYPE_KEYS`), searching for `entityId` using `Array.prototype.find`.
    *   This nested iteration is $O(\text{configs} \times \text{types} \times \text{entities\_per\_type})$, causing inefficient lookups, especially if there are many configs or entities.
    *   We can eliminate the nested iteration and the `.find` call for every single type by caching/indexing the entities into a `Map`. Since `currentRawConfigs` is a mutable array where items are reassigned (`currentRawConfigs[index] = config`), we can track changes either by keeping track of the array state or simply rebuilding an index whenever the config array is fetched if its "version" changed. However, there's no "version" on the array.
    *   Wait, the prompt suggests maintaining an index. A robust way to do this without worrying about tracking mutations on the array directly is to build a cache map *inside* `createConfigRoutes` that maps `entityId -> entity`. We can build it lazily and invalidate it inside the `POST /api/config/update` and `POST /api/config/add` routes. But what if the config changes from outside (e.g., file reloaded)?
    *   Since `getCurrentRawConfigs()` returns a reference to the array `currentRawConfigs` managed by `server.ts`, and `server.ts` does `currentRawConfigs = []` on reload, the array reference might change. Or its length or elements might change.
    *   A simple `WeakMap` from `rawConfig` object to an `EntityIndex` map, or caching an index that is rebuilt when `rawConfig` references change.
    *   Since the number of configs is usually small (e.g. 1-5 ports), we can map `rawConfig` -> `Map<string, entity>`.

    ```typescript
    const configEntityIndexes = new WeakMap<any, Map<string, any>>();

    // In the route
    if (type === 'entity') {
      for (const rawConfig of currentRawConfigs) {
        let index = configEntityIndexes.get(rawConfig);
        if (!index) {
          index = new Map<string, any>();
          for (const entityType of ENTITY_TYPE_KEYS) {
            const entities = rawConfig[entityType];
            if (Array.isArray(entities)) {
              for (const e of entities) {
                if (e && e.id) index.set(e.id, e);
              }
            }
          }
          configEntityIndexes.set(rawConfig, index);
        }
        foundEntity = index.get(entityId);
        if (foundEntity) break;
      }
    }
    ```
    *   If a `rawConfig` is mutated in place (e.g., `list[index] = newEntity`), the `WeakMap` wouldn't know!
    *   Let's check if `rawConfig` is mutated in place. In `POST /api/config/update`:
        ```typescript
        // 4. Update the original object structure with the modified data
        loadedYamlFromFile.homenet_bridge = normalizedFullConfig;

        // ... write new config ...

        // 6. Update in-memory raw config
        ctx.setCurrentRawConfigs(configIndex, normalizedFullConfig);
        ```
    *   `ctx.setCurrentRawConfigs` replaces the object at `configIndex` with `normalizedFullConfig`. It's a completely *new* object. The old `rawConfig` object is discarded, so its `WeakMap` entry will be garbage collected. This is perfect!

2.  **Verify Mutability:**
    *   Does `POST /api/config/add` also replace the whole config?
        ```typescript
        loadedYamlFromFile.homenet_bridge = normalizedFullConfig;
        // ...
        ctx.setCurrentRawConfigs(configIndex, normalizedFullConfig);
        ```
    *   Yes, it also replaces the object in the array with a new reference (`normalizedFullConfig`, generated by `normalizeConfig(loadedYamlFromFile.homenet_bridge)`).
    *   What about updates to the same `rawConfig` that don't go through `setCurrentRawConfigs`?
        *   Wait, `normalizedFullConfig` is the new reference, and `ctx.setCurrentRawConfigs(configIndex, normalizedFullConfig)` sets it.
    *   This means the `HomenetBridgeConfig` references in the `currentRawConfigs` array change whenever there's a modification! Thus, `WeakMap` mapping `rawConfig` (the `HomenetBridgeConfig` object) to a `Map<string, any>` is perfectly safe, fully dynamic, and correctly handles both config loading (new objects) and updates (replaced objects).
    *   This guarantees an $O(N)$ index build *once* per config lifetime, and $O(1)$ lookups thereafter.
    *   Total lookup time becomes $O(C)$ where $C$ is the number of configs (very small, usually 1).

    We can apply the same logic to `automation` and `script`! But the prompt focuses on `ENTITY_TYPE_KEYS` nested loop at line 60. Optimizing just the entity part is the core of the request, but doing it for automations/scripts is trivial if we just use `.find` since it's just one array (not a nested loop). The problem in entities is the nested loop over `ENTITY_TYPE_KEYS` ($O(K \times M)$ per config).

    Actually, let's keep the `WeakMap` at the module level in `config.routes.ts` or inside the closure of `createConfigRoutes`. Inside `createConfigRoutes` is better so it doesn't leak between different router instantiations (e.g. in tests).
